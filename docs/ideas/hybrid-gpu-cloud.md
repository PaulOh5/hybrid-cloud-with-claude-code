# 하이브리드 GPU 클라우드

> 작성일: 2026-04-23
> 상태: 방향성 확정, 검증 단계 진입 전

## Problem Statement

**How Might We** — 어떻게 하면 기존 워크스테이션 구매자와 자체 데이터센터의 GPU 자원을 **사용자 입장에선 구분 없는 단일 클라우드**로 통합하면서도, 제한된 개발 인력으로 6개월 내 한국 GPU 클라우드 시장에 진입할 수 있을까?

## Recommended Direction

**"DC 우선 출시 + Private Zone을 베타 당근으로 + 3개월 후 커뮤니티 노드 공개"** 3단 로드맵.

순수 마켓플레이스(🅐)는 6개월 안엔 정산·가용성·분쟁 해결이 엉성할 수밖에 없고, 첫 사고로 브랜드가 무너진다. 순수 Private Zone(🅑)은 "SSH 잘 쓰는데 왜 웹 UI?"라는 반문을 깰 독립적 가치가 부족하다. 따라서 두 방향의 가장 강한 부분만 취한다.

1. **기반(1–4개월):** 자체 DC 단일 zone의 GPU VM 클라우드로 상업 오픈 — compute-agent / main-api / ssh-proxy / frontend 4개 컴포넌트를 DC 환경에서만 먼저 검증.
2. **병행(2–4개월):** 기존 구매자 5–10명을 베타 파트너로 초청, 그들의 워크스테이션을 **Private Zone**으로 등록 — 자기 팀 전용으로만 사용, 공유는 없음. 이 단계의 목적은 매출이 아니라 **구매자 사용성 검증 + PR용 서사 축적**.
3. **확장(4–6개월):** "커뮤니티 노드" 토글 공개. Private Zone 소유자가 원하면 유휴 시간을 공용 풀에 기여하고 수익 쉐어. 이 시점에 비로소 **"단일 자원 풀" 하이브리드 서사**가 완성되어 2차 PR 소재가 된다.

이 순서의 핵심은 **"하이브리드"라는 기술 복잡도를 타임라인 뒤로 미루되 브랜드 약속은 처음부터 유지**한다는 점이다. DC 오픈이 먼저라 인지도 확보 목표는 1단계로 달성하고, 마켓플레이스 리스크는 3단계에서 **이미 검증된 베타 파트너 풀** 안에서만 열어 통제 가능하다.

## Key Assumptions to Validate

구현 착수 전에 반드시 확인해야 할 것들:

- [ ] **수익화 의향 가정** — 기존 워크스테이션 구매자 중 "내 GPU를 빌려주고 수익 얻고 싶다"는 사람이 의미 있는 비율(≥10%)로 존재. 검증: 구매자 30명 1:1 인터뷰, 플랫폼 착수 전.
- [ ] **국내 수요 가정** — 현재 RunPod/Vast.ai를 쓰는 한국 사용자가 "국내 저지연·원화 결제·한국어 지원" 이점으로 이탈할 가격 민감도. 검증: 한국 GPU 클라우드 수요자 10명 인터뷰 ("왜 해외 쓰나? 국내면 쓸래? 얼마까지?").
- [ ] **KVM VM 적합성 가정** — 학습·추론 워크로드 사용자가 Docker·bare-metal 대비 KVM VM에 만족. 검증: 베타 파트너가 실제 워크로드 최소 1주일 운영 후 NPS 측정.
- [ ] **DC 가동률 가정** — 워크스테이션 기반으로 지은 DC가 상용 SLA(≥99%)를 낼 수 있음. 검증: DC 오픈 전 48시간 스트레스 테스트 + 가상 장애 시뮬레이션.
- [ ] **GPU passthrough·분할 가정** — 판매 중인 워크스테이션의 GPU·메인보드·IOMMU 그룹 구성에서 여러 GPU를 서로 다른 VM에 vfio-pci로 동시 passthrough 가능. NVLink 페어가 있는 경우 페어 단위로 묶여 할당됨. 검증: 실제 하드웨어 1대에서 2 VM 동시 기동 후 각각 GPU 벤치마크 돌려 성능 손실·격리 확인.
- [ ] **VM 간 GPU 재할당 청결성 가정** — VM 종료 후 GPU가 다음 테넌트에 재할당될 때 메모리 잔존·상태 누수 없이 깨끗하게 리셋됨. 검증: 의도적으로 앞 VM에서 GPU 메모리에 마커를 남기고, 다음 VM에서 읽히지 않음을 확인.
- [ ] **SSH Proxy 확장성 가정** — 단일 ssh-proxy가 동시 SSH N(≥100) 세션에서 지연·누수 없이 동작. 검증: 부하 테스트 후 확장 지점 문서화.
- [ ] **"기존 구매자 커뮤니티" 규모 가정** — 베타 5–10명을 모을 수 있을 정도로 기존 구매자 풀이 활성·연락 가능. 검증: 구매 이력 DB로 세그먼트 규모 확인.

## MVP Scope (1–4개월, Phase 1)

> **핵심 기능 원칙:** 하나의 멀티 GPU 노드 안에서 **여러 VM이 GPU를 나눠 점유**한다. 이 플랫폼의 상품성은 "노드당 N개의 GPU를 1·2·4 단위로 쪼개 여러 사용자에게 판매"하는 능력에 달려 있으며, Phase 1의 필수 요건이다.


**포함(In):**
- compute-agent: 자체 DC의 각 GPU 노드에 설치. KVM VM 생성·삭제·상태 보고. **GPU passthrough(vfio-pci) 기반 멀티 GPU 분할 수용.** main-api와 영속 연결.
- **GPU 분할 — 정적 프로파일 방식:** 노드마다 관리자가 사전 정의한 레이아웃(예: 4 GPU 노드 → `1·1·1·1` / `2·2` / `4` 중 택일) 하에서 인스턴스 생성. 사용자는 "1·2·4 GPU" 같은 크기 옵션 중 선택.
- 인스턴스 ↔ GPU 매핑 테이블: main-api가 각 노드의 GPU 슬롯 점유 상태를 추적. 같은 노드 안에서 여러 VM이 공존.
- 멀티 GPU VM의 NVLink 페어링 존중: 프로파일 정의 시 물리 토폴로지(같은 NVLink 도메인·같은 PCIe 스위치) 고려한 프리셋만 노출.
- VM 종료 시 GPU 상태 초기화(메모리 클리어·리셋)로 테넌트 간 격리 보장.
- main-api: 인스턴스 라이프사이클, zone·노드 상태, GPU 슬롯 스케줄링, 사용자 인증, 과금 이벤트 로깅.
- ssh-proxy: 도메인 기반 SSH 라우팅 (`{random}.hybrid-cloud.com` → 해당 VM).
- frontend: 로그인, zone 목록, **GPU 크기 선택(1/2/4...) 기반 인스턴스 생성**/삭제, 접속 정보 표시.
- zone 모델: **단일 zone(본사 DC)만 존재**. 다중 zone UI는 있지만 데이터는 하나.
- 결제: 월정액 또는 선불 크레딧 중 하나 (시간당 초미세 과금은 v2). **GPU 개수별 요금 티어**는 v1에 포함.
- 운영 대시보드: 내부용, 노드 가동·GPU 슬롯 점유율·알림.

**제외(Out):**
- 개인 노드 등록 플로우 (Phase 2)
- 마켓플레이스 정산·수익 쉐어 (Phase 3)
- **동적 GPU 패킹 / 재배치 스케줄러** — 임의 크기 요청을 빈 슬롯에 자동 배치하거나 프래그먼트 병합. Phase 2.
- **MIG(NVIDIA Multi-Instance GPU) 서브 분할** — 하나의 GPU를 여러 인스턴스로 쪼개기. Phase 2+ (하드웨어·드라이버 매트릭스 복잡).
- 라이브 마이그레이션, 오토스케일링
- 컨테이너(Docker) 런타임 (KVM VM만)
- 다중 리전, 재해복구

## Not Doing (and Why)

- **진짜 마켓플레이스 (개방형 참여)** — 임의 GPU 소유자가 노드 제공. 이유: 신뢰·정산·분쟁·품질 편차가 2년 규모 프로젝트이며 인력 제약에 맞지 않음. 베타는 이미 아는 구매자로만.
- **"단일 풀" 컨셉을 1일차에 제공** — 이유: 기술 복잡도가 출시 리스크를 수용 불가능한 수준으로 밀어 올림. 3단계에서 **검증된 풀 안에서만** 열 것.
- **컨테이너(Docker/k8s) 지원** — 이유: KVM 단일 기술로 먼저 검증. 컨테이너는 격리·과금 모델·보안이 다른 문제를 만들며, 두 런타임 동시 지원은 초기 인력 범위 밖.
- **해외 고객·다국어 UI** — 이유: "국내 저지연·한국어"가 차별점. 해외 확장은 인지도 확보 후.
- **초단위 과금·Spot 시장 가격** — 이유: 정산 복잡도 증가. 월 정액/선불 크레딧으로 시작해 수요 신호 확보 후 정교화.
- **자체 비용의 대형 PR 캠페인** — 이유: 6개월 인지도 확보는 **베타 파트너 서사 + 커뮤니티 노드 공개**의 2단 PR로 충분. 유료 캠페인은 반응을 본 후.
- **동적 GPU 패킹 스케줄러** — 이유: Phase 1은 관리자가 미리 정의한 정적 프로파일(`1·1·1·1` / `2·2` / `4`) 안에서만 인스턴스를 만든다. 임의 크기를 빈 슬롯에 자동 배치·재조합하는 로직은 NUMA·NVLink·프래그먼테이션을 엮는 별개의 복잡도 축이며 Phase 2로 미룬다.
- **GPU 내 분할(MIG)** — 이유: 하나의 GPU를 여러 인스턴스가 공유하는 NVIDIA MIG 모드는 드라이버·카드·CUDA 버전 매트릭스와 엮여 운영 난도가 크다. Phase 1은 "GPU 단위 정수 할당"만. MIG는 Phase 2+.

## Open Questions

구현 착수 전 또는 초기 단계에서 결정해야 할 것들:

- **compute-agent가 NAT 뒤의 개인 홈 네트워크에서 동작하려면 어떤 터널링이 최선인가?** (reverse tunnel, WireGuard, Tailscale-like, 자체 TCP 터널) — Phase 2 착수 전 PoC 필요.
- **과금 모델: 월정액 / 선불 크레딧 / 시간당 중 어느 것으로 시작?** 결정은 국내 수요자 인터뷰 결과에 연동.
- **Phase 3 수익 쉐어 비율:** 구매자(노드 제공자) : 플랫폼 = ?. Vast.ai·Salad 벤치마크 필요.
- **"Zone" UX를 v1부터 노출하는가, 아니면 v1은 zone 개념 없이 "지역/노드" 정도로 단순화하는가?** — UI 복잡도 vs v2 이행 비용의 트레이드오프.
- **상품명·도메인:** `hybrid-cloud.com`은 예시인가 실제 확보했는가? 브랜드 네이밍이 인지도 확보의 입구.
- **베타 파트너 인센티브:** 크레딧, 수익 쉐어 선행, 지분 어떤 조합? — 모집 전 결정.
- **보안 경계:** 베타 파트너의 개인 워크스테이션이 Private Zone으로 동작할 때, 그들의 로컬 데이터와 VM 간 격리 수준을 어디까지 약속할지 — 법적·기술적 라인.
- **GPU 프로파일 정책:** 각 노드의 분할 레이아웃을 관리자가 고정하는가, 사용자가 요청 시점에 바꾸는가? Phase 1엔 관리자 고정 권장하나 UX 영향 검토 필요.
- **NVLink 보장 수준:** 멀티 GPU VM이 NVLink 페어 할당을 보장받는가? 가격 티어에서 "NVLink 보장" vs "일반"으로 구분할 가치가 있는가?
- **GPU 할당 최소 단위:** 1 GPU 미만(MIG·시분할) 인스턴스 티어가 사업상 필요한가? — 저가 수요 확보 vs 복잡도 증가 trade-off (현재 계획은 "아니오, Phase 2+").

## Appendix: 검토된 대안

다음은 Phase 1에서 생성하고 Phase 2에서 탈락시킨 방향들의 기록이다.

| 방향 | 핵심 | 탈락 사유 |
|------|------|-----------|
| 🅐 Owner Network 마켓플레이스 | 구매자 전용 닫힌 단일 풀 즉시 구성 | 정산·분쟁·SLA 복잡도, 첫 사고 시 브랜드 리스크 |
| 🅑 Private Zone 단독 | 구매자가 자기 팀용 사설 클라우드로만 사용 | 독립 가치가 약함 ("SSH로 잘 쓰는데 왜?") |
| Inversion (DC만, 워크스테이션은 로컬 개발환경) | 하이브리드 = 로컬·원격 연속성 | "내 GPU 공유"의 원래 비전 포기, 차별점 상실 |
| 10x 마켓플레이스 | 한국 내 모든 GPU 소유자 참여 | 본업과 이격, 마켓 운영은 전혀 다른 사업 |
| 극단 단순 MVP (하이브리드 제거) | 단일 zone 사설 클라우드 | 차별점 소멸, 일반 GPU 클라우드와 구별 불가 |
